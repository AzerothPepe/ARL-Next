# 🤖 AI Agent 核心准则

> 处理本项目任务时，必须**严格遵守**以下规范。

## 1. 核心设计原则
- **极简主义**: 保持克制，少即是多。
- **职责单一**: 每个模块或函数只做一件事。
- **清晰易懂**: 代码需自我解释。优先保证人类可读性，避免取巧写法。

## 2. 编码与设计规范
- **最少依赖**: 优先使用**标准库**，避免引入第三方包。
- **错误处理**: 必须显式处理错误，禁止生吞异常。
- **前端优化**: 更新 Nginx 配置必须开启 Gzip，防止大体积静态资源明文传输拖慢加载。
- **UI 最小化**: 最小化点击次数。优先扁平化布局（侧边栏、标签页），**避免**深层嵌套以降低交互成本。

## 3. AI 交互行为
- **强制中文**: 生成的所有工件 (Artifacts)**必须**使用中文。
- **放慢思考**: 放慢分析步伐，保持逻辑连贯，禁止跳跃式决策。
- **无痕执行**: 临时文件（脚本、测试数据等）在任务结束后**必须主动删除**，保持工作区整洁。
- **高危确认**: 销毁容器或清理网络等高危操作前，**必须询问**用户并获批。
- **精炼文档**: 更新 `Readme.md` 需内容充实、语言精炼，禁止废话。

## 4. 部署与 CI/CD
- **热修复易失**: 容器内“热修复”在执行 `start-prod.sh` 后会丢失（重新拉取镜像）。
- **代码修改与提交闭环**: 任何有效修复、优化、依赖或代码修改，必须同步修改本地源码。修改后必须先询问：“是否需打包热更新到服务器测试？”测试成功无误后，再询问是否执行 `git commit`。
- **凭证位置**: 服务器SSH登录信息存放在 `server.txt`。
- **部署前置检查**:
  1. 代码 Push 前，**必须询问**：“是否需打 Tag 触发镜像构建？”
  2. 触发 Github CI 前，**必须检查并修正 `Dockerfile.prod`**。
  3. 执行 `start-prod.sh` 前，**必须确保**代码已推送且最新镜像构建完成。

## 5. 代码审查方法论
执行代码审查时，必须遵守以下规范：
- **追溯完整调用链**: 禁止仅看局部逻辑。必须从最初入口完整追溯至最终出口（含沿途参数转换与下游消费逻辑）后再下结论。
- **核实多调用方**: 函数被多处调用时，必须检查所有调用点上下文，不可偏盖全。
